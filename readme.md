# LLM-Compiler

## 总体流程

1. 模型推理流程 -> 初始状态 $p_0$
2. $p_0$ -> 输入给 -> LLM-Compiler (MCTS + LLM)
3. LLM-Compiler -> 输出 -> Optimized Schedule (最佳调度)
4. Optimized Schedule -> 编译 -> GPU Machine Code

## MCTS部分

- **初始化**：直接将C语言形式的推理脚本作为 $p_0$，并附带一个算子调优状态说明
- **变换**：从根节点 $p_0$ 开始遍历，使用UCT公式来选择下一个进行变换的节点 $p_i$。
- **扩展**：LLM对某个算子进行某种类型的调优，得到新的状态 $p_{i+1}$
- **模拟**：从新节点 $p_{i+1}$ 开始，**随机应用一系列合法转换**，得到一个模拟的最终程序 $p_{sim}$，并使用一个**硬件成本模型**来预估该模拟程序的性能（长期性能）
- **反向传播**：将成本模型预测出的分数（奖励 $W$）反向传回给该节点的所有祖先节点，同时更新路径上每个节点的访问次数 ($N$) 和累积奖励 ($W$) 

### 扩展

LLM 接收包含**当前计算图状态、父节点计算图状态、祖父节点计算图状态**及其**对应的转换历史**和性能分数的 Prompt 。计算图状态指的是计算图中各个节点目前已被实施的优化种类、参数；节点指蒙特卡洛树搜索中的节点。

prompt：要求 LLM 分析不同变体间的差异，识别性能变化的来源，并基于硬件成本模型生成新的转换建议。LLM 输出的转换建议会被解析和过滤，以确保其在有效转换集合中。

根据转换建议，生成新的状态 $p_{i+1}$ 的计算图状态，并将其作为 $p_{i+1}$ 的子节点。

## 最终状态

当最终收敛到一个较优的计算图状态时，使用LLM对相应算子的ninetoothed kernel进行重写，生成最终的优化代码。
