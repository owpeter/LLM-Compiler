# LLM-Compiler

## 总体流程

指定算子的基础实现（Naive Code/Schedule）
-> 初始状态 $p_0$
-> LLM-Compiler (MCTS + LLM)
-> 算子的最优调度参数/最优代码
-> 编译为 GPU Machine Code。

## MCTS部分

- **初始化**：将算子的基础实现作为 $p_0$
- **变换**：从根节点 $p_0$ 开始遍历，使用UCT公式来选择下一个进行变换的节点 $p_i$。
- **扩展**：LLM对算子进行某种调优，调优种类和方向根据LLM的先验知识进行自主选择，得到新的状态 $p_{i+1}$
- **模拟**：从新节点 $p_{i+1}$ 开始，**随机应用一系列合法转换**，得到一个模拟的最终程序 $p_{sim}$，并使用一个**硬件成本模型**来预估该模拟程序的性能（长期性能）
- **反向传播**：将成本模型预测出的分数（奖励 $W$）反向传回给该节点的所有祖先节点，同时更新路径上每个节点的访问次数 ($N$) 和累积奖励 ($W$)

### 扩展

1. $p_i$，当前算子的代码/调优配置文件，以及性能分数
2. 父节点与祖父节点的算子代码/调优配置文件，以及性能分数
3. *硬件瓶颈分析：如果可能，提供当前实现的静态分析数据（如：计算密集度、内存带宽占用预估、寄存器溢出压力等）*。

prompt：要求 LLM 分析不同变体间的差异，识别性能变化的来源，并生成新的调优建议。LLM 输出的调优建议会被解析和过滤，以确保其在有效调优集合中。

根据调优建议，生成新的状态 $p_{i+1}$ 的算子代码/调优配置文件，并将其作为 $p_{i+1}$ 的子节点。

## 最终状态

当最终收敛到一个较优的计算图状态时，根据配置文件使用LLM对相应算子的ninetoothed kernel进行重写，生成最终的优化代码。
