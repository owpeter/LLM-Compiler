# 基于 ninetoothed 的算子调优方案说明

## 1. 项目定位

这个项目的核心目标，是为推理中的关键算子自动找到更优的执行方案，并将最终结果落到 `ninetoothed` 生成的高性能 kernel 上。

从根目录 [readme.md](/Users/wangtianyi/Desktop/study/LLM-Compiler/readme.md) 可以看出，项目整体思路不是单纯依赖人工写 kernel，也不是只做通用编译优化，而是把两件事情结合起来：

1. 用 `MCTS + LLM` 在较大的调优空间中搜索更优的算子配置
2. 用 `ninetoothed` 把最终配置固化为可执行的高性能实现

因此，这个项目的本质可以概括为：

**以智能搜索决定“该怎么调”，以 ninetoothed 完成“调优结果如何落地执行”。**

## 2. 技术路线

## 2.1 总体路线

项目采用的是一条“搜索驱动 + 生成落地”的技术路线。

整体流程可以概括为：

1. 为某个目标算子给出初始实现或初始调度状态
2. 结合 workload、硬件特征和可调参数空间，构造优化问题
3. 使用 MCTS 在候选空间中进行多轮搜索
4. 在每次扩展阶段，由 LLM 基于上下文生成新的调优建议
5. 使用硬件成本模型对候选方案进行快速评估
6. 逐步收敛到更优的 schedule 或 kernel 参数组合
7. 将最优参数映射到 `ninetoothed` 的 kernel 生成体系
8. 生成最终可运行的高性能算子实现

这个过程把“找最优配置”和“生成最终代码”明确分成了两个阶段。前者强调搜索效率和决策质量，后者强调性能落地和工程可执行性。

## 2.2 搜索阶段：MCTS + LLM 联合决策

项目在搜索阶段采用了 MCTS 与 LLM 协同的方式。

其中：

- `MCTS` 负责维护搜索树，决定当前应该优先探索哪个候选方向
- `LLM` 负责理解已有变体之间的差异，并提出下一步调优建议

这种分工非常关键。因为算子调优空间通常具有两个特点：

1. 候选参数多，组合空间大
2. 参数之间存在较强耦合，靠穷举或纯随机搜索效率较低

MCTS 的作用，是在“继续深入已有好方向”和“探索新方向”之间做平衡；LLM 的作用，则是利用先验知识做更有针对性的扩展，而不是盲目生成新候选。

换句话说，这条路线不是让 LLM 单独承担搜索任务，而是把 LLM 放在一个受约束、可反馈、可迭代的搜索框架中，使其输出更稳定、更可利用。

## 2.3 评估阶段：成本模型加速搜索闭环

如果每生成一个候选方案都进行真实编译和真实测量，搜索成本会非常高，迭代速度也会很慢。为了解决这个问题，项目引入了硬件成本模型。

在这一阶段，系统会对候选 schedule 或参数组合进行快速性能预测，并将预测结果作为搜索奖励回传给 MCTS。

这样做的意义在于：

1. 避免每轮都进行高成本实测
2. 提高单位时间内可探索的候选数量
3. 让搜索过程形成“生成建议 -> 快速评估 -> 更新策略”的闭环

因此，成本模型在这条路线中承担的是“搜索加速器”的角色。它不直接生成优化方案，但能显著提升整个调优流程的效率。

## 2.4 落地阶段：通过 ninetoothed 固化优化结果

当搜索过程收敛到较优的调优结果后，项目不会把结果停留在“参数建议”层面，而是继续通过 `ninetoothed` 将其转化为实际可执行的 kernel。

这里的关键思想是：

- 搜索阶段负责找到更优的实现方向
- ninetoothed 负责把这一方向表达成具体的 kernel 组织方式

从仓库中的设计文档可以看出，`ninetoothed` 侧可承载的调优内容主要包括：

- tile 大小
- warp 数量
- pipeline stage 数
- 不同算子特有的 block 划分方式与执行形态

这意味着，前面搜索得到的 schedule，并不是抽象的“建议”，而是能够进一步映射为 `ninetoothed` 中可执行的实现参数。

因此，`ninetoothed` 在整个系统中扮演的是“性能落地层”的角色。它把搜索得到的高层优化决策，转化为具体可运行的算子实现。

## 2.5 技术闭环总结

从方法论上看，这个项目形成了一条比较完整的技术闭环：

1. 从初始算子实现出发
2. 构造可搜索的调优空间
3. 用 MCTS 控制搜索过程
4. 用 LLM 生成有方向性的候选变体
5. 用成本模型做快速性能评估
6. 用 ninetoothed 将最终结果固化为高性能 kernel

这条路线的重点，不在于单一技术本身，而在于几种能力之间形成了协作：

- MCTS 提供搜索框架
- LLM 提供知识驱动的扩展能力
- 成本模型提供快速反馈
- ninetoothed 提供高性能落地能力

## 3. 技术优势

## 3.1 相比人工调优，更易扩展

传统算子优化往往依赖专家手工分析和逐步试参。这种方式在少量核心算子上有效，但很难扩展到更多算子、更多硬件和更多 workload。

本项目的优势在于，它把算子调优过程转化为一个可复用的自动化流程。也就是说：

- 不再完全依赖人工反复试错
- 可以将经验沉淀为搜索空间、提示模板和成本模型
- 更容易迁移到新的算子类型和新的问题规模

这使得系统具备更强的可扩展性。

## 3.2 相比纯穷举搜索，更高效

如果单纯依赖穷举或网格搜索，当参数维度增多时，搜索代价会快速膨胀。

本项目通过 `MCTS + 成本模型` 规避了这一问题：

- MCTS 不必遍历所有组合，而是优先探索更有潜力的分支
- 成本模型减少了真实测量的次数
- LLM 提高了候选生成的方向性

因此，系统并不是靠“试得更多”来找到好解，而是靠“试得更聪明”来提升效率。

## 3.3 相比纯 LLM 搜索，更稳定

如果只让 LLM 直接生成调优结果，通常会面临几个问题：

- 输出不稳定
- 容易偏离合法参数空间
- 缺少长期反馈，难以持续改进

而在这个项目中，LLM 不是独立工作，而是被放在一个带有约束和反馈的搜索框架中：

- 候选会被限制在可用参数空间内
- 每一步都有性能反馈
- 历史节点信息会影响下一步扩展

这样做的好处是，LLM 的能力被保留下来了，但随意性被显著降低，整体调优流程更加稳健。

## 3.4 相比只做搜索，更能真正落地

很多优化方案停留在“找到了一个更优参数组合”，但并没有把结果变成一个可持续复用的高性能实现。

本项目的优势在于，它不仅能搜索到候选方案，还能借助 `ninetoothed` 将结果进一步转化为真正的 kernel 实现。

这意味着项目不是停留在“理论上更优”，而是具备了从优化决策到执行代码的完整路径。这样的方案更容易服务于真实推理系统，而不只是做离线分析。

## 3.5 更适合异构硬件与多样化 workload

仓库中的相关文档反复强调了硬件感知和 workload 感知的重要性。对于不同硬件、不同 batch/seq/head 规模，算子的最优执行形态往往并不相同。

本项目的路线天然适合处理这类问题，因为它的输入本身就包含：

- 硬件信息
- workload 信息
- 可调参数空间

这使得系统可以围绕具体运行场景做有针对性的搜索，而不是依赖单一的静态最优配置。

从技术视角看，这种能力意味着项目更接近“面向场景的自适应调优”，而不是“一次调优、处处适用”的固定策略。

## 3.6 兼具研究表达与工程价值

这条路线还有一个明显优势，就是研究逻辑和工程逻辑比较统一。

从研究表达上，它有清晰的方法结构：

- 搜索问题建模
- LLM 辅助扩展
- 成本模型评估
- kernel 生成落地

从工程价值上，它又具备实际可执行性：

- 能对接具体算子
- 能沉淀为可复用的调优流程
- 能输出最终可运行的实现

因此，它既适合讲“自动化算子调优”的技术故事，也具备进一步走向真实推理系统的基础。

## 4. 一句话总结

如果用一句话概括这个项目的技术路线与优势，可以表述为：

**该项目通过“MCTS + LLM + 成本模型”搜索算子的更优调度，再利用 ninetoothed 将优化结果固化为高性能 kernel，从而实现兼顾搜索效率、调优质量与工程落地能力的自动化算子调优方案。**
